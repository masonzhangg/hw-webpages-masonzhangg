Overview

In this homework, I built a physically-based path tracer from the ground up. Starting with ray generation, I implemented ray-triangle and ray-sphere intersection using the Moller–Trumbore algorithm and quadratic formula respectively. To make rendering tractable on complex scenes, I constructed a BVH that reduced intersection complexity from O(n) to O(log n), bringing render times from nearly a minute down to a fraction of a second. With the acceleration structure in place, I implemented direct lighting via both uniform hemisphere sampling and importance sampling, learning firsthand how targeted sampling dramatically reduces variance. I then extended this to full global illumination using recursive path tracing with Russian Roulette termination, which produces physically correct indirect lighting effects like color bleeding and soft fill light in shadowed regions that are simply impossible to replicate faithfully in a rasterizer. Finally, I implemented adaptive sampling, which uses a statistical convergence criterion to concentrate samples in high-variance regions and avoid wasting computation on pixels that have already converged.

Part 1: Ray Generation and Scene Intersection

Camera Ray Generation: I generate a ray for each pixel. Given the normalized image coordinates, I first map them onto the virtual sensor place in the camera space, which is at z = -1. The sensor spans both horizontally and vertically, so the mapping is: x = (2x - 1) * tan(hFov/2), y = (2y - 1) * tan(vFov/2), z = -1. The camera -space direction vector is then rotated into the world space using the c2w matrix and normalized. The resulting ray originates at the camera world position with the minimum t = nClip, and the maximum t = fClip.

Pixel Sampling: For each pixel (x, y), I generate ns_aa rays by jittering the sample point within the pixel using a uniform grid sampler. Each jittered sample is normalized by image dimentsions before being used in the generate_ray function. The final fixel color is the average of est_radiance_global_illumination, evaluated over all sameples.

Primitive Intersection: For every primitive, has_intersection does a boolean test, while intersect populates an Intersection struct. The maximum t is updated to t on each valid intersection, so any subsequent intersection test for the same ray automatically rejects geometry that is further away. This is so I find the nearest intersction.

Moller-Trumbore solves for the intersection and the barycentric coordinates. A point on the triangle can be written as: P = (1 - u - v)*p1 + u*p2 + v*p3. So if I set this equal to the ray (P = o * td). I get [-d | (p2-p1) | (p3-p1)] * [t, u, v]^T = o - p1. Then, I applied Cramer's rule and this yields closed-form solutions This intersection is only valid if the ray is not parallel to the triangle, the hit point is inside the triangle, and the hit is within the ray's active range. If valid, the surface normal is computed by interpolating the three vertex normals using the barycentric weights (w, u, v) where w = 1 - u - v. This gives smooth shading across the triangle surface.

Part 2: Bounding Volume Hierarchy

The BVH is built recursively. At each call, I compute the bounding box that encloses all primitives in the current range and create a new node with that box. If the number is at or below the maximum leaf size, I mark the node as a leaf by setting its iterators to return immediately. For interior nodes, I need to partition the primitives into two groups. In the axis selection step, I compute the extent of the bounding box along each axis and split along the longest axis. This is because splitting along the longest dimension produces the most spatially separated children, so both subtrees are not hit by the same ray. In the split point step, I use the mean of the primitive centroids along the chosen axis as the split value. Each primitive whose centroid falls below this split value goes to the left child and the rest goes to the right. There is also an edge case. If all centroids fall on one side of the same mean, the paritition produces one empty group. This is detected by checking if middle = start or middle = end. In this case, I fall back to a simple median split, so both children have atleast one primitive.

The performance improvement from BVH acceleration is dramatic. In Part 1, rendering the bunny (CBbunny.dae) with only brute-force intersection took roughly 40-45 seconds, since every ray had to be tested against every primitive, giving O(n) intersection cost per ray. With BVH acceleration, the same scene rendered in under a second, well under 0.2 seconds in practice. This reflects the reduction to O(log n) average intersection cost, since the BVH lets us prune entire subtrees of geometry the moment a ray misses their bounding box.

Part 3: Direct Illumination

Uniform Hemisphere Sampling: Direct lighting is estimated by sampling random directions uniformly across the hemisphere at the hit point. For each of the num_samples iterations, I draw a random object-space direction from the hemisphere sampler, transform it to world space, and cast a shadow ray from hit+p in that direction with minimum t = EPS_F to avoid self-intersection. If the shadow ray hits something and that thing has non-zero emission, we apply the Monte-Carlo estimator of the reflection equation. The key weakness of this approach is that most sampled directions will miss all light sources entirely, making it very inefficient and noisy at low sample counts.

Importance Sampling: Wasted samples are eliminated by sampling directions that are guarenteed to point toward a light source. For each light in the scene, we call sample_L which returns the emitted radiancem a world space direction toward the light, the distance to the light, and the pdf of that sample. We trasform wi_world to object space and skip the sample if cos(theta) = wi_obj.z <= 0 which means that the light is behind the surface. Else, we cast a shadow ray. If nothing blocks the shadow ray, we apply the reflection equation. For point lights, we only draw 1 sample since all samples are the same. The result for each light is averaged over its sample count before being accumulated into L_out.

Importance sampling produces cleaner results than the unifrom hemisphere sampling at the same sample count. With hemisphere sampling, only the small fraction of rays that happen to hit a light contribute signal. This results in a grainy appearance. Important sampling directs every sample towards a light source so all samples carry useful information and the variance is lower. The difference is most visible at low sample counts, hemisphere sampling produces a hard-to-see noisy image while importance sampling produces a mostly-correct result. The one visual difference is that hemisphere sampling captures the soft glow around the area light itself while importance sampling's shadow rays do not.

Part 4: Global Illumination

First, at_least_one_bounce_radiance evaluates one_bounce_radiance at the current hit point. It then checks whether recursion should continue. If so, the russian roulette is applied with a continuation probability of 0.65. If the coin flip fails, terminate, if it succeeds, we call isect.bsdf->sample_f to importance-sample a new incoming direction wi_obj in the object space, transform it to the world space, and construct a new ray from hit_p with depth - 1. If this new ray hits the scene, we recurse into at_least_one_bounce_radiance and accumulate the result weighted by f * cos_theta / (pdf * rr_prob). The isAccumBounces=false, isolates individual bounces by only adding the direct term when the ray is at exactly max_ray_depth, letting the recursion reach the desired depth without accumulating shallower bounces.

The 2nd bounce image shows light that has reflected off two surfaces before reaching the camera. The most visually striking effect here is color bleeding. The 3rd bounce is dimmer still, but it continues to fill in areas that the 2nd bounce left dark. Shadows become softer and the overall scene feels more uniformly lit. The color bleeding effects introduced at the 2nd bounce are slightly reinforced as colored light continues to propagate through the scene. By this point, each successive bounce contributes diminishing returns in terms of brightness, but the cumulative effect of adding the 2nd and 3rd bounces together is a scene that feels physically grounded and naturally lit. This is where path tracing fundamentally differs from rasterization. A rasterizer computes lighting purely as a local operation. Techniques like ambient occlusion, screen-space reflections, and baked lightmaps are all approximations that attempt to fake what path tracing computes exactly. The 2nd and 3rd bounces in particular capture effects that are essentially impossible to reproduce correctly in real-time rasterization without expensive precomputation.

Part 5: Adaptive Sampling

Adaptive sampling: Standard path trcing uses a fixed number of samples per pixel regardless of how visually complex each pixel is. This is wasteful. Adaptive sampling addresses this by checking each pixel's convergence as samples accumulate and stopping early once the estimate is stable enough, concentration computation where it is needed instead. The convergence criterion is based on a 95% confidence interval. After taking n samples with illuminance values x1, x2, ..., xn, we track two running sums. s1 = Σxk and s2 = Σxk2 which let us compute the sample mean and variance without storing every individual sample. I is the half-width of the 95% confidence interval around μ. When I ≤ maxTolerance × μ (default tolerance 0.05), we are 95% confident the true pixel value lies within 5% of our current estimate, and we stop sampling.

The implementation replaces the fixed loop in raytrace_pixel with a while loop that runs up to ns_aa samples. We use Vector3D::illum() to collapse the three-channel radiance into a single scalar illuminance value for the statistics, since we want one convergence decision per pixel rather than one per channel. The convergence check is only performed every samplesPerBatch samples. Finally, sampleCountBuffer is written with the actual n rather than ns_aa, which produces the sample rate visualization image where red pixels indicate regions that required many samples and the blue pixels indicate regions that converged quickly.
